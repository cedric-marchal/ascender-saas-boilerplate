# Codebase Audit: security (full codebase)

First pass (single-endpoint OWASP lenses) found a strong posture with no critical issue. The **combinatorial deep-dive** (second pass, see the dedicated section below) then found one **critical** chained bug — a paywall bypass on project mutation actions — plus several state-machine and race findings. Every row was re-verified against the source before inclusion.

- **Date**: 2026-07-06
- **Scope**: full codebase (app/, features/, lib/, proxy.ts, next.config.ts, prisma config)
- **Health**: fair (one isolated, addressable critical in the deep-dive section)
- **Findings (first pass)**: 0 critical, 2 warning, 3 minor
- **Findings (deep-dive)**: 1 critical, 4 warning, 3 minor

Health: `good` = no critical findings; `fair` = critical findings exist but are isolated and addressable; `poor` = systemic or widespread critical findings.

## Deep-dive: combinatorial findings (2026-07-06)

Second pass targeting bugs that only appear by **chaining features, racing requests, or reordering events** — not single-endpoint issues. Four adversarial angles: race/TOCTOU, business-logic state machines, cross-tenant cache/session, Stripe webhook ordering. Every row below was re-verified in source by the auditor (not just the sub-agents).

| 🔴  | security | `features/projects/actions/{create,update,delete}-project.action.ts:10`                                                       | **Paywall bypass / broken write authorization.** The "projects" pro feature is gated only on the read page (`dashboard/projects/page.tsx:49` `requireCustomerPlan("pro")`). | 🟡  | security | `features/billing/services/stripe/handle-webhook.service.ts:289-315` vs `:337-386`                                            | **Out-of-order webhook re-grants entitlement.** No `event.created`/monotonicity guard; the subscription write is last-write-wins. Residual (documented, low): stale-`updated`-overwrites-newer-`updated` (both non-deleted) is still last-write-wins on status/price — self-heals on the next correct event; a full monotonicity guard would need an `event.created` column.

Not vulnerabilities (verified clean, recorded so they aren't re-investigated): cross-tenant cache bleed — all Redis keys are org/event-scoped from server-derived ids, `"use cache"` is only the intentionally-global admin dashboard, React `cache()` usages are per-request and correctly keyed; `activeOrganizationId` is never client-trusted; invitation email-match, email-change-vs-verified gate, active-org switching, and admin→owner self-promotion are all protected (see webhook agent's "guards that are correct": checkout.session.completed grants nothing, price + status allowlists enforced at read, id-based customer mapping). Webhook idempotency is a documented check-then-set (effects are idempotent). Seat-cap concurrent-accept relies on Better Auth's `membershipLimit` for race-safety — library-internal, medium confidence, backstopped once the `@@unique` above is added.

### Verified protections (no findings — recorded so future audits don't re-litigate)

- **Authn/authz layering**: `proxy.ts` edge redirect + per-layout and per-page guards (`requireAdmin`, `requireCustomer*`, `lib/session.ts:126-134`); safe-action clients correctly tiered (`lib/safe-action.ts:55-123`); every action enumerated and using the appropriate client; `role`/`slug` fields `input: false` in `lib/auth.ts:151-160` (no self-elevation); org role `owner` cannot be minted via invite/change schemas.
- **IDOR**: all customer/org services filter by `userId` or verified membership in the Prisma `where`; `organizationId` derived from session, never client input; project/billing/audit-log services re-verify membership; admin services only reachable from `requireAdmin`-gated pages.
- **Stripe webhook**: signature verified via `constructEvent` (`handle-webhook.service.ts:179`), raw body, Redis idempotency set only after success, 5xx on transient errors so Stripe retries.
- **Avatar upload**: authenticated, rate-limited, Zod size/MIME check plus byte-level re-validation via sharp, server-generated storage key (no path traversal), `startsWith` guard on deletion.
- **Rate limiting fails closed**: Upstash env vars required at boot (`lib/env.ts:36-37`); `checkRatelimit` propagates Redis errors; IP extraction throws in production when absent (`utils/ratelimit/get-request-identifier.ts:30-32`). Sensitive unauthenticated endpoints (sign-in/up, password reset, contact, invitations) all covered.
- **Secrets & env**: no hardcoded secrets in git-tracked files (CI/test placeholders only); t3-env separation clean; no server module imported into client bundles.
- **Headers/CSP**: nonce-based CSP with `strict-dynamic`, X-Frame-Options DENY, HSTS preload, nosniff, Permissions-Policy (`proxy.ts:18-49`).
- **Injection/XSS**: no raw SQL/eval/child_process; all `dangerouslySetInnerHTML` usages are static JSON-LD without user input; contact form goes through Resend JSON API (no header injection), input Zod-validated.
- **Data exposure**: explicit `select` everywhere on sensitive models; no password hashes, tokens, or `stripeCustomerId` returned to non-admin surfaces.
