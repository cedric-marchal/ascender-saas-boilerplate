# i18n & Routing Rules — next-intl, pathnames, locales

## Context

Rules for internationalized routing, translations, and locale handling. Applies to `src/i18n/**`, `src/messages/*.json`, `src/app/**`, and any file producing links, user-facing strings, or locale-dependent values.

## Source of Truth (P0)

`src/i18n/routing.ts` (the `pathnames` map) is the single source of truth for ALL routes and internal links.

- Every route MUST be declared in `pathnames`. English slugs are the canonical internal keys; localized slugs (e.g. French) exist ONLY as `pathnames` values.
- Route folders under `src/app/[locale]/` use the English canonical names (`sign-in/`, `pricing/`, `dashboard/billing/`) — never localized folder names.
- All navigation goes through the typed wrappers from `@/i18n/navigation` (`Link`, `redirect`, `usePathname`, `useRouter`, `getPathname`). Outside a Next request context (emails, boundaries), use `getStaticPathname` from `@/i18n/get-static-pathname` — it reads the same `pathnames` map.
- NEVER hand-build a URL string for an internal route.

```tsx
// ✅ Correct
import { Link } from "@/i18n/navigation";

<Link href="/pricing">...</Link>;

// ✅ Correct (outside request context)
const url = `${BASE_URL}${getStaticPathname("/pricing", locale)}`;

// ❌ Wrong: hand-built, bypasses pathnames
<a href={`/${locale}/tarifs`}>...</a>;
```

## No Binary Locale Logic (P0)

`locale === "fr" ? ... : ...` is FORBIDDEN everywhere.

- Locale-dependent values (BCP47 `inLanguage`, `og:locale`, hreflang) live in `src/i18n/locale-metadata.constant.ts`, keyed `satisfies Record<Locale, LocaleMetadata>` — the typecheck fails when a locale is added without its metadata.
- hreflang/alternates maps are generated by iterating `routing.locales` (see `src/i18n/get-locale-alternates.ts`), never enumerated.
- Adding a locale must only touch: `routing.locales`, `LOCALE_METADATA`, `src/messages/<locale>.json`.

## Translations (P0)

- Server components/metadata: `getTranslations`. Client components: `useTranslations`.
- Zod schemas store translation KEYS (`"validation.email.required"`); resolve at display time via `src/utils/errors/translate-field-errors.ts`. Server-thrown `AppError` messages are keys translated at the boundary (`translate-app-error.ts` in `handleServerError`/`handleApiError`).
- `Record<Enum, string>` label constants hold translation keys, resolved at render.
- Catalogs `src/messages/en.json` / `src/messages/fr.json` MUST stay in key parity — `__tests__/i18n/messages-parity.test.ts` enforces it. Add every new key to BOTH files.
- Every page/layout under `src/app/[locale]/` MUST call `setRequestLocale(locale)` — omitting it silently degrades static rendering in production.

## Context-Free Boundaries (P0)

Some surfaces render OUTSIDE `NextIntlClientProvider` / the `[locale]` request config and MUST NOT use `useTranslations`, `getTranslations`, or the `@/i18n/navigation` `Link` (they crash with "No intl context found"):

1. **Root boundaries** (`src/app/not-found.tsx`, `src/app/forbidden.tsx`, `src/app/unauthorized.tsx`, `src/app/maintenance/page.tsx`): derive locale from the `NEXT_LOCALE` cookie (`getLocaleFromCookies`), translate via `getTranslator` (`src/i18n/get-translator.ts`), link via plain `next/link` + `getStaticPathname(...) as Route`. Their shared page components take a `locale: Locale` prop.
2. **Error boundaries** (`error-page.tsx`, `global-error-page.tsx` — client, provider may be unmounted): locale via `getClientLocale` (document.cookie), strings from the minimal `src/i18n/error-boundary-messages.ts` (never import the full catalogs into client error bundles).
3. **Emails** (React Email templates, rendered detached from any request): `locale` prop + `getTranslator` + `getStaticPathname`.
4. **API routes / Better Auth hooks** (excluded from locale negotiation by `src/proxy.ts`): `getLocaleFromCookies` / `getLocaleFromRequest`.

`router.refresh()`-only usages may import `useRouter` from `next/navigation` (no URL produced, nothing to localize).

## Anti-Patterns (NEVER)

```
locale === "fr" ? x : y                       // binary locale logic
<a href={`/${locale}/...`}>                    // hand-built internal URL
useTranslations in a root boundary/email      // no provider → crash
new route folder without a pathnames entry    // unroutable via typed nav
key added to en.json but not fr.json          // parity test failure
page under [locale] without setRequestLocale  // silent dynamic rendering
```
